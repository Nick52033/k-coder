# k-Coder 运行时架构

## 目标

k-Coder 是桌面编程智能体，不是给 Shell 套一层聊天界面。运行时必须提供确定性的工具分发、工作区隔离、可恢复对话、取消、审计能力，以及与模型供应商无关的统一事件。

初始架构有意把 Tauri 保持在系统边界，确保智能体运行时无需打开桌面窗口也能独立测试。

## 系统结构

```text
React 界面
  -> 类型化 Tauri 命令
  -> 应用服务
  -> AgentRuntime
       -> ContextBuilder（上下文构建）
       -> Provider（模型适配）
       -> ToolRegistry（工具注册）
       -> PolicyEngine（权限策略）
       -> ToolExecutor（工具执行）
       -> ThreadRepository（会话存储）
  -> 版本化 AgentEvent 事件流
  -> Zustand 展示状态
```

## Rust 模块职责

| 模块 | 负责 | 不得负责 |
| --- | --- | --- |
| `commands` | Tauri 请求校验和响应映射 | 智能体循环、SQL、模型响应解析 |
| `agent` | Turn 生命周期、上下文组装、循环控制、取消 | 具体工具和 HTTP 协议格式 |
| `providers` | 认证头、请求体、SSE 解析 | 工作区访问和界面事件 |
| `tools` | 工具定义、参数校验、执行结果 | 用户授权决策 |
| `policy` | 允许、需要审批、拒绝决策 | 工具实现 |
| `storage` | JSONL 事件日志、SQLite 索引、Repository | 界面状态 |
| `protocol` | 版本化命令和事件 | 业务行为 |

## 前端职责

```text
src/
  components/       通用视觉组件
  features/         对话、设置、工作区、活动面板
  stores/           从运行时事件派生的展示状态
  api/              类型化 invoke/listen 封装
  types/            前端协议类型
```

前端不得直接调用模型 API、打开 SQLite、执行命令或决定某项工具操作是否安全。

## Turn 状态机

```text
queued（排队）
  -> streaming（流式响应）
  -> awaiting_approval（等待审批）
  -> running_tool（执行工具）
  -> streaming（继续响应）
  -> completed（完成）

任意活动状态 -> cancelled（已取消）
任意活动状态 -> failed（失败）
```

运行时必须先持久化会改变状态的事件，再发布对应的界面事件。这样应用崩溃后可以恢复状态，界面也能重新构建。

## 智能体循环

1. 校验会话、工作区、模式和用户输入。
2. 追加用户消息和 `turn_started` 事件。
3. 从持久化历史、项目规则和 Token 预算构建上下文。
4. 携带原生工具 Schema 发起流式 Provider 请求。
5. 将文本增量作为非持久化展示事件转发。
6. 使用注册的 Schema 校验每个工具调用。
7. 请求 `PolicyEngine` 给出允许、审批或拒绝决策。
8. 执行已授权工具，把结构化结果追加到历史中。
9. 继续调用模型，直到模型返回最终文本或达到运行时限制。
10. 持久化 `turn_completed`、`turn_failed` 或 `turn_cancelled`。

## Provider 契约

Provider 负责把外部协议转换为少量内部事件：

- `text_delta`：文本增量
- `tool_call_delta` 或完整工具调用
- `usage`：Token 用量
- `completed`：请求完成
- 类型化 Provider 错误

`AgentRuntime` 不得直接解析 OpenAI、Anthropic 或 Google 的协议载荷。当前适配器支持 OpenAI-compatible Chat Completions、OpenAI Responses、Anthropic Messages 和 Google Gemini；新增供应商必须继续通过独立适配器接入，不能在智能体循环内部增加协议分支。

## Phase 2 只读运行时边界

- 应用启动时把进程启动目录 canonicalize 为本次工作区根目录；正式的工作区选择和切换由 Phase 6 提供。
- `list_directory` 只列直接子项，默认最多 200 项、最多 500 项，并忽略 `.git`、`node_modules`、`target`、`dist` 和 `build`。
- `read_file` 识别 UTF-8 BOM、UTF-16LE 和 UTF-16BE；单文件最大 4 MiB，单次最多返回 256 KiB。
- 单个 Turn 最多执行 8 次 Provider 迭代、24 次工具调用；同名同参调用最多重复 2 次。
- `assistant_tool_calls` 与每个 `tool_result` 先写入 JSONL，再发布工具生命周期界面事件。

## Phase 3 安全编辑边界

- 模型只能通过注册的 `apply_patch` 或 `write_file` 提议文件修改；回复文本和 Markdown 代码块没有写入能力。
- `apply_patch` 使用严格的 Begin/End、文件头和 hunk 语法，支持新增、修改、删除和移动；`write_file` 只用于明确的完整 UTF-8 文件替换。
- 写工具的模型参数不包含审批字段。`PolicyEngine` 按读取、写入、删除和外部访问分类，后端创建并持有一次性审批能力。
- 审批请求先写入 JSONL，再发布到界面。用户可以选择文件、编辑 Patch、查看统一或并排 Diff，并明确应用或拒绝；未批准、超时和取消都不会写盘。
- 执行前重新解析 Patch、规范化路径并验证审阅时的 SHA-256。文件在审阅期间发生改变时返回冲突，不覆盖新内容。
- 多文件操作在单个编辑锁内执行；任一操作失败会根据修改前快照逆序回滚已经开始的操作。
- `change_applied` 保存有界的修改前后快照和完整 Diff。撤销前验证修改后哈希，防止撤销覆盖应用后产生的新修改，并记录 `change_undone` 审计事件。
- 当前限制为单个 Patch 1 MiB、最多 20 个文件、单文件 512 KiB、单次变更前后快照合计 2 MiB。

## 工具契约

每个工具必须提供：

- 稳定的名称和说明
- JSON 输入 Schema
- 类型化参数校验
- 风险等级
- 异步处理器
- 包含成功状态、输出和元数据的结构化结果
- 取消行为
- 对有效参数、无效参数、拒绝、取消和边界情况的单元测试

`ToolRegistry` 负责发现 Schema 和处理器；`PolicyEngine` 负责授权；`ToolExecutor` 负责协调策略判断、审批、超时、取消、审计事件和处理器执行。

## 工作区安全

文件操作必须按以下顺序处理路径：

1. 把相对路径拼接到已选择的工作区。
2. 规范化路径，并对最近存在的父级目录执行 canonicalize。
3. 在平台允许时解析符号链接和目录联接。
4. 使用当前平台对应的路径比较规则。
5. 拒绝规范化后位于工作区之外的路径。

普通工作区工具不接受绝对路径。访问工作区之外的资源必须使用独立的显式 Capability，并经过用户审批。

不能因为 Shell 的当前目录位于工作区内就认为命令天然安全。命令执行前必须经过策略判断，并支持输出上限、超时、取消和进程树终止。

## 持久化

JSONL 是对话和 Turn 事件的持久化事实来源。SQLite 保存便于查询的投影，例如会话列表、设置、用量和搜索索引。

```text
runtime-data/
  sessions/<thread-id>.jsonl
  k-coder.db
  logs/
```

在可行情况下，SQLite 数据应当可以从 JSONL 重新构建。API Key 必须存入操作系统凭据设施，不能写入数据库或 TOML 配置文件。

## 子智能体

子智能体是使用受限上下文、预算、工具和工作区权限的普通 `AgentRuntime` 实例。它们复用相同的 Provider、工具、策略、事件和取消契约。在评估证明更深层委派确有价值之前，最大派生深度限制为一层。

## 初始版本不包含的能力

- 手机远程控制
- 机器人工作流设计器
- 浏览器录制与回放
- 自主长期记忆
- 插件市场
- 首发即支持多种 Provider 协议

这些能力保留在路线图中作为后续选项，但不作为成熟编程智能体的必要前置条件。
