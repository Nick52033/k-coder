# ADR 0002：可审阅的文件变更事务

- 状态：已接受
- 日期：2026-07-19

## 背景

编程智能体需要修改多个文件，但模型输出、用户审阅和实际写盘发生在不同时间。直接执行模型生成的文本会产生路径逃逸、静默覆盖、半完成变更和无法审计的问题。后续的命令、MCP、Skills 与机器人也需要复用同一套授权原则。

## 决策

文件修改采用“提议、审阅、提交、审计”四段流程：

1. 模型只能调用注册的 `apply_patch` 或 `write_file`，不能从回复文本触发写入。
2. `patch` 独立领域模块严格解析 Patch，并在读取工作区后生成逐文件预览、SHA-256 和完整 Diff。
3. `PolicyEngine` 对写入和删除返回需要审批。审批请求 ID 由后端生成，一次性审批通道在事件发布前注册；模型参数不能携带审批或哈希能力。
4. 用户批准时提交所选文件及审阅哈希。编辑过的 Patch 必须重新生成预览，后端再次解析并校验审批范围。
5. 写盘前再次校验规范化路径和修改前哈希。多文件变更在进程内编辑锁中顺序执行，失败时根据修改前快照逆序回滚。
6. 成功后持久化 `change_applied`，保存大小受限的前后快照和 Diff。撤销仅在当前内容仍匹配修改后哈希时执行，并持久化 `change_undone`。

## 影响

收益：

- 用户看到的 Diff 与批准范围是明确的结构化数据。
- 拒绝、超时、哈希冲突和路径拒绝都在写盘前结束。
- 多文件失败可以恢复，成功变更可以审计和安全撤销。
- 后续外部工具可以沿用后端持有权限、模型只提交意图的边界。

成本与限制：

- JSONL 会保存快照，因此必须限制 Patch、文件数量、单文件和快照总大小。
- 当前只处理文本文件，不保留二进制内容或文件元数据。
- 事务保证进程内失败回滚，不等同于跨进程崩溃或断电时的文件系统原子事务。
- 撤销只针对单个变更集，不自动重排相互依赖的多次变更。

## 未采用方案

### 从 Markdown 代码块提取并写盘

未采用，因为自由文本缺少稳定语法、审批范围、冲突哈希和可审计工具边界。

### 让模型传入 `approved: true`

未采用，因为权限不能由请求权限的一方自行声明。审批能力只能由后端创建并通过用户操作解析。

### 只依赖 Git 回滚

未采用，因为工作区可能没有 Git、可能包含未提交修改，而且整仓回滚会覆盖不属于当前变更集的用户内容。
