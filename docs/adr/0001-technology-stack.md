# ADR 0001：技术选型

- 状态：已接受
- 日期：2026-07-18

## 背景

k-Coder 需要桌面界面、本地文件和进程访问、长时间模型流式请求、持久化对话状态，以及严格的工作区访问边界。如果先用 Node.js 完成全部运行时、以后再迁移到 Rust，就需要重写系统中风险最高的部分。

## 决策

项目从一开始使用以下技术栈：

| 层级 | 选择 |
| --- | --- |
| 桌面框架 | Tauri 2 |
| 运行时 | Rust 2024 edition |
| 异步与 HTTP | Tokio + reqwest |
| 序列化 | serde + serde_json |
| 持久化存储 | JSONL + SQLite |
| 前端 | React + TypeScript + Vite |
| 界面状态 | Zustand |
| 包管理器 | pnpm |
| 运行时错误 | thiserror |

React 负责渲染和用户交互；Rust 负责模型、工具、策略、进程和持久化行为。

## 影响

收益：

- 不需要规划运行时重写。
- Provider、工具、存储和生命周期状态之间有强类型约束。
- 更容易控制子进程、取消和桌面打包。
- 桌面分发体积通常小于内置 Electron 运行时的方案。

成本：

- 第一版开发速度会慢于把所有逻辑放在单个 Electron 主进程文件中。
- Rust 异步和所有权边界需要谨慎设计接口。
- Provider 与 MCP 的现有示例通常更偏向 JavaScript 生态。

应对方式是控制第一版范围并坚持模块边界，而不是维护一套临时运行时。

## 未采用方案

### 临时使用 Electron + Node.js 运行时

未采用，因为后续迁移需要重写工具执行、取消、存储、MCP 进程管理和并发调度。

### 永久使用 Electron + Node.js

该方案适合快速产品原型，但 k-Coder 更重视本地进程安全和长期运行时结构，因此不采用。

### 直接复制 cn-codex

未采用，因为它的功能范围和大型核心文件会在智能体循环可以独立测试之前引入大量复杂度。
